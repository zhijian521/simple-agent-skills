# 通用代码规范

核心原则：**简单、简洁、合理、统一**。

## 决策顺序

开始写代码前，按下面顺序判断：

1. 能否在现有文件内完成
2. 能否用更直接的实现完成
3. 是否真的需要新增抽象、工具函数、组件、hook、composable、store
4. 是否能让后续维护者一眼看懂

如果新文件、新抽象、新目录没有明显收益，优先不创建。

## 代码风格

- 缩进使用 `4` 个空格
- 使用单引号，模板字符串使用反引号
- 语句末尾保留分号
- 多行对象、数组、参数列表保留尾逗号
- 文件末尾保留一个空行
- 单行长度尽量不超过 `120`，最多不超过 `160`

## 复杂度控制

- 优先提前返回，减少嵌套
- `return` 后不再跟 `else`
- 单个函数只做一件事
- 嵌套超过 `3` 层时优先重构
- 避免为了一致性引入无收益的抽象
- 无未使用的变量、函数、类型、导入
- 无注释掉的废弃代码

正例：

- 页面逻辑能在一个文件内清楚完成，就不拆成 4 个文件
- 一个方法能顺着写完，就不要包成多层中间调用

反例：

- 为了“解耦”把简单请求拆成 service -> manager -> adapter -> mapper
- 为了“复用”把只用一次的逻辑抽成公共方法

## 命名

- 变量、函数：`camelCase`
- 组件、类、类型、接口：`PascalCase`
- 常量：`UPPER_SNAKE_CASE`
- 文件名：`kebab-case`
- 布尔值：使用 `is`、`has`、`can`、`should` 前缀

## 注释

- 默认少写注释，优先写自解释代码
- 注释用于说明意图、边界、约束，不复述代码表面行为
- 仅对对外暴露且不够直观的函数补充 JSDoc
- 复杂文件允许使用分区注释，但只标结构，不写废话

以下场景必须补充注释：

- 对外导出且副作用明显的方法
- 关键配置对象
- 特殊兼容逻辑
- 复杂字段转换逻辑

分区注释格式：

- 模板：`<!-- 区块名 -->`
- 脚本：`/*== 区块名 ==*/`
- 样式：`/*== 区块名 ==*/`

## 导入

导入顺序固定为：

1. 第三方依赖
2. 项目别名导入，如 `@/`
3. 相对路径导入

补充要求：

- 每组之间空一行
- `type` 导入放在各组末尾
- 不使用 `import * as`
- 不保留未使用导入

## 错误处理

- 异步逻辑默认使用 `async/await`
- 可能失败的异步流程要有错误处理
- `catch` 不能为空
- 返回外部数据时先做空值或结构判断，再使用
- 用户可感知的失败要给出明确反馈

补充要求：

- 在入口处校验关键结构，不要在业务代码里到处零散兜底
- 不要把错误处理写成一层层 `?.` 和 `??`

## 配置与常量

- 接口地址、key、超时、分页默认值、业务枚举统一抽离
- 配置优先放在 `config`、`constants`、`env` 或项目既有集中位置
- 业务文件里不直接写死可复用配置

反例：

```ts
localStorage.setItem('token', token);
request('/api/user/list');
const pageSize = 20;
```

正例：

```ts
import { STORAGE_KEYS } from '@/constants/storage';
import { API_PATHS } from '@/constants/api';
import { DEFAULT_PAGE_SIZE } from '@/constants/pagination';
```

## 提交信息

- 提交信息使用清晰、可检索、单一职责的写法
- 一个提交只表达一类改动，不把无关修改揉在一起
- 默认格式：`<type>(<scope>): <summary>`
- `summary` 使用简洁中文，直接说明本次改动结果

常用 `type`：

- `feat`
- `fix`
- `refactor`
- `perf`
- `style`
- `docs`
- `test`
- `build`
- `chore`

## 禁止项

- 过度设计
- 无意义封装
- 单次复用也强行抽公共层
- 逻辑散落在多个文件来回跳转
- 用规范掩盖糟糕设计
- 接口字段逐项补默认值
- 配置和 key 写死在业务文件
