# 反模式压制规则

这个文件优先级最高，用来压制 Agent 在项目开发中最常见的坏习惯。

只要没有明确收益或明确业务需求，默认不要做下面这些事。

## 1. 不要无必要新增文件

- 能在现有文件内完成，就不要新建文件
- 能在当前目录完成，就不要额外加目录层级
- 只有出现明确复用、职责分离、文件过大时才拆文件

反例：

- 为一个简单页面新增 `api/`、`hooks/`、`constants/`、`types/`、`utils/`、`components/`
- 为一个一次性逻辑单独建 `use-xxx.ts`

正例：

- 先在当前页面或当前模块内完成
- 等复用关系或复杂度真的出现再拆

## 2. 不要为了“安全”堆无意义空判断

- 不要每访问一层数据都机械加 `?.`
- 不要把明显必有的数据也写成多层兜底
- 不要因为害怕报错，就写大量没有业务价值的防御判断

反例：

```ts
const name = data?.user?.profile?.name ?? '';
const list = res?.data?.list ?? [];
const total = res?.data?.total ?? 0;
```

如果接口契约本来就明确，这种写法通常只是掩盖问题。

正例：

- 只校验关键边界
- 明确哪一层是可信的，哪一层是不可信的
- 真正不可信的数据在入口统一校验

## 3. 不要接口返回后逐字段补默认值

- 不要把接口对象每个字段都重新组装一遍
- 不要为了“稳”把每个字符串都补 `''`、每个数组都补 `[]`
- 只有在展示层或转换层有明确需求时才补默认值

反例：

```ts
const user = {
    id: res.id ?? '',
    name: res.name ?? '',
    mobile: res.mobile ?? '',
    avatar: res.avatar ?? '',
};
```

正例：

- 直接使用接口数据
- 只在页面展示时对少数字段兜底
- 只有在需要转换字段结构时才组装新对象

## 4. 不要无必要抽公共层

- 只调用一次的逻辑不要抽公共函数
- 只在一个页面使用的组件不要抽全局组件
- 只在一个文件内使用的常量不要急着抽公共常量文件
- 不要为了“看起来规范”增加中间层

## 5. 不要把配置和常量写死在业务文件

这些内容应集中管理：

- 接口 base URL
- storage key
- cache key
- 路由常量
- magic string
- magic number
- 超时时间、分页默认值、限制数量

反例：

```ts
localStorage.setItem('token', token);
request('/api/user/list');
const timeout = 30000;
```

正例：

- 统一放到 `constants`、`config`、`env` 或项目已有配置文件

## 6. 不要写废话注释，也不要遗漏关键注释

- 不要注释“获取列表”“设置值”这种代码表面行为
- 关键配置、关键方法、复杂转换、特殊兼容必须有说明

## 7. 不要让结构复杂化

- 不要为了抽象而抽象
- 不要为了统一形式牺牲可读性
- 不要把简单流程拆成多个文件来回跳转

## 8. 默认优先的做法

- 先少建文件
- 先少建目录
- 先少加判断
- 先少做转换
- 先直接实现
- 再按真实复杂度演进
