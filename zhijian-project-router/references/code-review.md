# Code Review 规范

## 审查目标

优先发现：

1. 正确性问题
2. 潜在回归
3. 安全问题
4. 性能风险
5. 可维护性问题
6. 缺失测试或验证

如果没有明确问题，要明确写出“未发现明确问题”，但仍补充剩余风险和未验证项。

## 输出顺序

审查结果必须按下面顺序输出：

1. `Findings`
2. `Open Questions / Assumptions`
3. `Summary`

## Findings 写法

每条 finding 必须包含：

- 严重级别
- 文件路径
- 尽量精确的行号或定位
- 问题是什么
- 影响是什么
- 建议如何修复

## 审查检查项

### 通用

- 明显 bug 或边界遗漏
- 无收益抽象或过度拆分
- 死代码、未使用导入、无意义中间层
- 命名、结构、职责是否清晰
- 错误处理、空值处理是否完整

### TypeScript / JavaScript

- 是否滥用 `any`、`as`、非空断言
- 是否把同步、异步、异常分支混成一团

### Vue

- 是否把模板计算写得过重
- 是否错误使用 `watch`
- 是否不必要地引入 composable 或 store

### 小程序 / uni-app

- 是否滥用平台专属 API
- 是否缺少用户可感知的失败提示
- 是否存在跨端兼容风险

### 性能

- 是否有本可并行却串行的请求
- 是否在大列表、大对象、频繁渲染路径中做重计算

## 审查原则

- 只报告真实值得修的问题
- 结论要具体
- 不能只说“不好”，要说明风险与修法
