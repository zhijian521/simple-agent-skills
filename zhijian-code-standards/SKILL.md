# 通用代码规范

用于约束代码本身的质量，不负责规定最终回复格式。回复格式由 `zhijian-output-contract` 负责。

核心原则：**简单、简洁、合理、统一**。

## 1. 决策顺序

开始写代码前，按下面顺序判断：

1. 能否在现有文件内完成
2. 能否用更直接的实现完成
3. 是否真的需要新增抽象、工具函数、组件、hook、composable、store
4. 是否能让后续维护者一眼看懂

如果新文件、新抽象、新目录没有明显收益，优先不创建。

## 2. 代码风格

- 缩进使用 `4` 个空格
- 使用单引号，模板字符串使用反引号
- 语句末尾保留分号
- 多行对象、数组、参数列表保留尾逗号
- 文件末尾保留一个空行
- 单行长度尽量不超过 `120`，最多不超过 `160`

## 3. 复杂度控制

- 优先提前返回，减少嵌套
- `return` 后不再跟 `else`
- 单个函数只做一件事
- 嵌套超过 `3` 层时优先重构
- 避免为了一致性引入无收益的抽象
- 无未使用的变量、函数、类型、导入
- 无注释掉的废弃代码

可以直接内联的逻辑，不要拆成只被调用一次的包装函数。

## 4. 命名

- 变量、函数：`camelCase`
- 组件、类、类型、接口：`PascalCase`
- 常量：`UPPER_SNAKE_CASE`
- 文件名：`kebab-case`
- 布尔值：使用 `is`、`has`、`can`、`should` 前缀

命名要表达业务含义，避免 `data`、`item`、`temp`、`handle` 这类过宽名称泛滥。

## 5. 注释

- 默认少写注释，优先写自解释代码
- 注释用于说明意图、边界、约束，不复述代码表面行为
- 仅对对外暴露且不够直观的函数补充 JSDoc
- 复杂文件允许使用分区注释，但只标结构，不写废话

分区注释格式：

- 模板：`<!-- 区块名 -->`
- 脚本：`/*== 区块名 ==*/`
- 样式：`/*== 区块名 ==*/`

## 6. 导入

导入顺序固定为：

1. 第三方依赖
2. 项目别名导入，如 `@/`
3. 相对路径导入

补充要求：

- 每组之间空一行
- `type` 导入放在各组末尾
- 不使用 `import * as`
- 不保留未使用导入

## 7. 错误处理

- 异步逻辑默认使用 `async/await`
- 可能失败的异步流程要有错误处理
- `catch` 不能为空
- 返回外部数据时先做空值或结构判断，再使用
- 用户可感知的失败要给出明确反馈

## 8. 样式

- 类名简短、语义化
- 优先使用 Flex 和 Grid
- 避免无意义嵌套选择器
- 重复样式优先抽变量、混入或批量生成
- 不为了“炫技”使用难维护写法

## 9. 提交信息

- 提交信息使用清晰、可检索、单一职责的写法
- 一个提交只表达一类改动，不把无关修改揉在一起
- 默认格式：`<type>(<scope>): <summary>`
- `summary` 使用简洁中文，直接说明本次改动结果

常用 `type`：

- `feat`：新增功能
- `fix`：修复问题
- `refactor`：重构代码，不改变外部行为
- `perf`：性能优化
- `style`：纯样式或格式调整，不改逻辑
- `docs`：文档更新
- `test`：测试补充或调整
- `build`：构建、脚本、依赖、工程配置调整
- `chore`：杂项维护

补充要求：

- `scope` 写模块、页面、组件、服务名，保持稳定，不要过宽
- 不写空泛摘要，如“优化代码”“调整内容”“修改问题”
- 不混用中英夹杂废话，保持一句话可读

示例：

```text
feat(auth): 增加短信验证码登录
fix(order-list): 修复分页切换后重复请求
refactor(user-form): 简化提交流程并收敛校验逻辑
docs(skills): 补充提交流程说明
```

## 10. 禁止项

- 过度设计
- 无意义封装
- 单次复用也强行抽公共层
- 逻辑散落在多个文件来回跳转
- 用规范掩盖糟糕设计
