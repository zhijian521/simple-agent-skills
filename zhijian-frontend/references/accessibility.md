# 可访问性规范

以 WCAG 2.2 AA 作为通用参考基线，优先使用原生 HTML 语义，并结合项目组件库和目标平台验证。自动扫描只能发现部分问题，不能代替键盘和辅助技术测试。

## 语义与名称

- 使用原生 `button`、`a`、`input`、`dialog` 等元素，不用 `div` 模拟已有控件。
- 每个交互控件必须有可访问名称；表单优先使用可见且关联的 `label`。
- 非装饰图片提供描述目的的 `alt`；装饰图片使用空 `alt`。
- 标题层级、landmark 和页面 `lang` 应反映真实结构。
- 只有原生语义不足时使用 ARIA，并同步维护 name、role、value 和状态属性。

## 键盘与焦点

- 所有功能必须可通过键盘完成，不依赖鼠标、悬停或复杂手势。
- 焦点顺序符合视觉和操作逻辑，焦点样式清晰可见，不使用无替代的 `outline: none`。
- 指针目标至少满足 WCAG 2.2 AA 的 24×24 CSS 像素或相应间距例外；触屏主操作优先采用更宽松的项目设计尺寸。
- 弹窗打开后将焦点移入，交互期间限制在合理范围，关闭后恢复到触发元素。
- 不制造键盘陷阱；菜单、Tab、列表框等复杂组件遵循 WAI-ARIA APG 的对应键盘模式。

## 视觉与动态内容

- 普通文本对比度至少 4.5:1，大文本至少 3:1；非文本控件和焦点指示器至少 3:1。
- 内容在 200% 文本缩放和常见窄视口下仍可阅读、操作，不无故截断关键信息。
- 不只用颜色、位置、声音或动画表达状态。
- 非必要动画响应 `prefers-reduced-motion`；闪烁内容不得造成风险。
- Toast、异步结果和校验状态等动态信息使用合适的 `aria-live` 或状态语义，避免重复播报。

## 表单与错误

- 提供清晰说明、自动填充属性和输入目的；placeholder 不替代 label。
- 错误以文本说明原因和修复方式，并通过 `aria-describedby` 等方式关联到字段。
- 提交失败后将焦点引导到错误摘要或首个错误字段，但不要打断正在输入的用户。

## 验证

- 先运行项目已有 Axe、Lighthouse、eslint-plugin-jsx-a11y 或同类检查。
- 手工使用 Tab、Shift+Tab、Enter、Space、方向键和 Escape 验证关键流程。
- 关键组件至少用目标平台的主流屏幕阅读器验证一次；无法执行时明确记录未验证风险。
- 修复后重新验证，避免焦点、键盘或语义回归。
