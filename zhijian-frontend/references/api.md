# API 规范

## 项目优先

先识别项目现有的响应格式、鉴权方式、校验库、请求封装和 API 文档约定。复用已有 helper 与 wrapper，不因本规范改造接口风格。

## 信任边界

- 在路由、Controller 或 Handler 入口校验外部输入的类型、必填项、格式和权限。
- 业务层仍校验自身关键前提，不盲目信任上层调用。
- 将外部数据转换为内部模型时，只转换确有语义或展示需求的字段，不逐字段补默认值。

## 鉴权与错误

- 公开、已认证和受权限保护的接口边界要明确。
- 服务端按资源和操作执行授权，不能只校验“是否登录”或依赖前端角色判断。
- 重复的鉴权、响应构造和错误映射优先复用项目已有 wrapper 或 helper。
- 保留 HTTP 状态码语义；错误响应不得包含堆栈、SQL、密钥或其他内部实现细节。
- 用户可感知的失败提供明确、可行动的提示。

## 写操作与资源保护

- Cookie 会话支持写操作时评估 CSRF 防护。
- 重试可能重复执行的写操作时评估幂等键、唯一约束或业务去重。
- 登录、验证码、上传、采集和昂贵查询等入口按风险设置限流、大小限制和超时。
- SQL、命令、模板或查询条件使用参数化 API；服务端请求外部 URL 时限制协议、目标与重定向。

## 契约与文档

- 新增、删除或改变 API 行为时，同步更新项目已有的接口文档、类型或契约测试。
- 项目已有 Route 注释、OpenAPI、Markdown 索引或文档检查脚本时，在同一变更中保持同步。
- 分页、排序、筛选和错误格式遵循项目既有约定；没有既有约定时再定义最小、清晰且可扩展的格式。

## 禁止项

- 不在业务文件散落重复的请求、鉴权或错误处理模板。
- 不依赖仅前端的校验作为安全边界。
- 不为尚不存在的 API 预建 client、目录或文档空壳。
