# 测试规范

## 先识别能力

- 优先运行和扩展仓库已有的测试、Lint、类型检查与统一校验命令。
- 不引入测试框架、覆盖率门槛或不可执行脚本，除非任务明确要求且项目确有需要。
- 修复缺陷时，优先补能复现该问题的回归测试。

## 测试范围

- 纯函数覆盖正常输入与关键边界。
- 组件测试关注用户可见行为、语义和键盘交互，不测试内部状态或实现细节。
- 集成测试覆盖模块之间的重要契约和数据流。
- 端到端测试只覆盖核心业务路径；工具选择遵循项目既有配置。

## 查询与断言

- UI 测试优先按 `role`、`label`、可见文本和 `alt` 查询；`data-testid` 作为最后选择。
- 异步测试等待可观察结果，不依赖固定延时。
- 断言外部可观察的结果，避免对样式类名、私有方法或框架实现做脆弱断言。

## 完成标准

- 运行与本次改动相关、且仓库可用的校验命令。
- 未运行的校验要在交付时明确说明原因。
- 不要求每个简单展示文件机械增加测试；优先覆盖风险高、行为复杂或曾发生缺陷的逻辑。
