# Code Review

这个 skill 负责统一审查输出，不负责生成代码。

## 1. 审查目标

优先发现：

1. 正确性问题
2. 潜在回归
3. 安全问题
4. 性能风险
5. 可维护性问题
6. 缺失测试或验证

如果没有明确问题，要明确写出“未发现明确问题”，但仍补充剩余风险和未验证项。

## 2. 输出顺序

审查结果必须按下面顺序输出：

1. `Findings`
2. `Open Questions / Assumptions`
3. `Summary`

`Findings` 永远放最前面，不能先写总结。

## 3. Findings 写法

每条 finding 必须包含：

- 严重级别：`Critical` / `High` / `Medium` / `Low`
- 文件路径
- 尽量精确的行号或定位
- 问题是什么
- 会造成什么影响
- 建议如何修复

按严重级别从高到低排序；同级按影响面排序。

## 4. Findings 模板

```markdown
## Findings

1. **High** [path/to/file.ts:42]
   问题：这里在接口失败时直接返回成功态，导致页面展示错误数据。
   影响：用户会看到旧数据，且无法感知刷新失败。
   建议：补充失败分支处理，并区分旧数据回显与本次请求结果。
```

如果没有问题，使用：

```markdown
## Findings

未发现明确问题。
```

## 5. 审查检查项

### 通用

- 是否有明显 bug 或边界遗漏
- 是否引入无收益抽象或过度拆分
- 是否存在死代码、未使用导入、无意义中间层
- 命名、结构、职责是否清晰
- 错误处理、空值处理是否完整

### TypeScript / JavaScript

- 是否滥用 `any`、`as`、非空断言
- 是否把可推导状态写得过重
- 是否把同步、异步、异常分支混成一团

### Vue

- 是否把模板计算写得过重
- 是否错误使用 `watch`
- 是否不必要地引入 composable 或 store
- 是否修改 props 或破坏单向数据流

### 小程序 / uni-app

- 是否滥用平台专属 API
- 是否缺少用户可感知的失败提示
- 是否存在跨端兼容风险
- 是否把页面逻辑、组件逻辑、接口逻辑耦合过深

### 性能

- 是否有本可并行却串行的请求
- 是否在大列表、大对象、频繁渲染路径中做重计算
- 是否存在明显重复请求、重复渲染、无缓存读取

### 测试与验证

- 是否缺少关键路径验证
- 是否遗漏错误路径、空数据、权限、兼容性验证

## 6. 审查原则

- 只报告真实值得修的问题，不为凑数挑刺
- 结论要具体，不写空泛评价
- 不能只说“不好”，要说明风险与修法
- 如果信息不足，写进 `Open Questions / Assumptions`
