# 反模式压制规则

优先级最高。只要没有明确收益或明确业务需求，默认不做下面这些事。

## 1. 不要无必要新增文件

- 能在现有文件内完成就不新建文件
- 能在当前目录完成就不额外加目录层级
- 只在出现明确复用、职责分离、文件过大时才拆文件

反例：为一个简单页面新建 `api/`、`hooks/`、`constants/`、`types/`、`utils/`、`components/`

正例：先在当前页面或模块内完成，复用关系或复杂度真正出现后再拆

## 2. 不要为了"安全"堆无意义空判断

- 不机械给每层数据访问加 `?.`
- 不把明显必有的数据写成多层兜底
- 不因为害怕报错写大量无业务价值的防御判断

反例：
```ts
const name = data?.user?.profile?.name ?? '';
const list = res?.data?.list ?? [];
```
如果接口契约明确，这种写法只是掩盖问题。

正例：只校验关键边界，真正不可信的数据在入口统一校验。

## 3. 不要接口返回后逐字段补默认值

- 不把接口对象每个字段重新组装一遍
- 不为了"稳"把每个字符串补 `''`、每个数组补 `[]`
- 只在展示层或转换层有明确需求时才补

反例：
```ts
const user = { id: res.id ?? '', name: res.name ?? '', mobile: res.mobile ?? '' };
```
正例：直接使用接口数据，页面展示时对少数字段兜底。

## 4. 不要无必要抽公共层

- 只调用一次的逻辑不抽公共函数
- 只在一个页面用的组件不抽全局组件
- 只在一个文件用的常量不急着抽公共常量文件
- 不为了"看起来规范"增加中间层
- 简单页面不拆成很多中间包装组件或工具函数

## 5. 不要保留死代码和无效结构

- 删除未使用的变量、函数、类型、导入
- 删除注释掉的废弃代码
- 删除模板中已不用的 class、样式中无人引用的选择器
- 删除无结构收益的包裹层
- 删除重复声明、无意义兜底、无意义中间变量

改动功能时顺手清掉同文件里的死代码和旧样式。结构能平铺就平铺。

## 6. 不要把配置和常量写死在业务文件

以下内容集中管理：
- 接口 base URL
- storage key / cache key
- 路由常量
- magic string / magic number
- 超时时间、分页默认值

反例：
```ts
localStorage.setItem('token', token);
request('/api/user/list');
const timeout = 30000;
```
正例：放到 `constants`、`config`、`env` 或项目既有集中位置。

## 7. 不要写废话注释，也不要遗漏关键注释

- 不写"获取列表""设置值"这种复述代码表面行为的注释
- 关键配置、关键方法、复杂转换、特殊兼容必须有说明
- 注释默认用中文，`TODO`/`FIXME`/`NOTE` 等标记可保留英文

## 8. 不要让结构复杂化

- 不为了抽象而抽象
- 不为了统一形式牺牲可读性
- 不把简单流程拆成多个文件来回跳转
- 不把简单页面写成多层条件分支、多层容器、多层样式修饰

## 9. 默认优先

先少建文件 → 先少建目录 → 先少加判断 → 先少做转换 → 先直接实现 → 先删冗余 → 先让模板、脚本、样式保持直白 → 再按真实复杂度演进

## 10. 渐进收敛

老项目不要求一次性改完，每次改到相关模块时顺手对齐规范。临时方案（workaround）不得长期留在线上。复制旧代码后只改局部变量名、保留无用冗余结构——不给过。
