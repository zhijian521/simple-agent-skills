# 小程序 / uni-app 规范

本 skill 只写小程序与跨端框架的增量规则。通用规则沿用 `zhijian-code-standards`，语言规则沿用 `zhijian-typescript-standards`。

## 1. 使用边界

- 原生微信小程序：使用本规范中的原生小程序规则
- `uni-app`：使用本规范中的跨端规则
- 如果是标准 Vue Web 项目，不要套用本 skill

## 2. 目录原则

- 页面目录按业务页面组织
- 组件目录只放明确复用组件
- `api/`、`utils/`、`styles/` 按需创建
- 静态资源目录命名保持项目已有习惯，常见为 `static/`、`images/`

不要预建一堆空目录。

## 3. 原生小程序页面

- `onLoad` 中处理入参归一化和首屏请求
- `setData` 只提交真正变化的字段
- 异步流程必须处理失败分支
- 用户可见失败优先使用明确提示，如 `showToast`
- 避免把复杂业务逻辑直接堆在 `Page({})` 内，必要时抽到同目录模块

## 4. 原生小程序组件

- `properties` 明确 `type` 与默认值
- 事件名保持简单稳定，如 `tap`、`change`、`submit`
- 自定义事件参数结构保持固定，不同场景不要忽左忽右
- 组件只暴露必要属性和事件

## 5. uni-app 规则

- 优先使用 `uni.*` API，不直接写平台专属 API
- 条件编译是最后手段，不作为默认解法
- 页面结构按项目已有方式组织，避免把 Web Vue 习惯硬搬进跨端项目
- 跨端样式优先考虑不同端的兼容表现，不追求某一端最花哨写法

## 6. 生命周期与状态

- 生命周期里只做当前阶段必须做的事
- 页面加载态、提交态、空态、失败态要清晰
- 多请求互不依赖时优先并行
- 避免多个地方维护同一份可推导状态

## 7. 样式与兼容

- 单位、布局、组件标签遵循项目既有技术栈
- 涉及跨端差异时，先判断是否真有差异，再决定是否条件编译
- 避免依赖平台私有行为实现核心业务

## 8. 常见误区

- 为了复用过早抽公共组件
- 在 `uni-app` 中直接沿用 `wx.*`
- 页面、组件、接口、工具函数拆得过碎
- 只顾跑通微信端，忽略其他端行为差异
